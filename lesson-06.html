REST APIs

What Is a REST API?

A REST API is a server that accepts HTTP requests and returns structured responses (usually JSON).

HTTP Methods GET — read data
POST — create new resource
PUT / PATCH — update
DELETE — remove
Status Codes 200 OK   201 Created
400 Bad Request
401 Unauthorised
404 Not Found   500 Server Error 
# Request
POST /users
Content-Type: application/json

{
  "username": "Username",
  "password": "Password123",
  "email": "student@example.com"
}

# Success response  →  200 OK
{ "userId": "1", "message": "Valid User" }

# Failure response  →  400 Bad Request
{ "error": "Invalid User" }

// app.js — validators are injected, not hard-coded
function createApp(validateUsername, validatePassword, validateEmail) {
  const app = express();
  app.post('/users', async (req, res) => {
    const { username, password, email } = req.body;
    const valid = validateUsername(username)
               && validatePassword(password)
               && validateEmail(email);        // ← called on every request
    if (valid) {
      res.send({ userId: '1', message: 'Valid User' });
    } else {
      res.status(400).send({ error: 'Invalid User' });
    }
  });
  return app;
}
The validators are passed in as arguments — this is Dependency Injection. It makes them swappable in tests.
Integration Testing

Testing HTTP APIs with Supertest

Supertest lets you send real HTTP requests to your Express app inside a Jest test — no browser, no running server.

What Supertest does:
  • Starts your Express app in memory
  • Sends a real HTTP request to it
  • Returns the response for you to assert on
  • Shuts down after the test — no cleanup needed
Install: npm install --save-dev supertest 
const request = require('supertest');
const createApp = require('./app');
const validateUsername = require('./validation/validateUsername');
const validatePassword = require('./validation/validatePassword');
const validateEmail  = require('./validation/validateEmail');

const app = createApp(
  validateUsername,
  validatePassword,
  validateEmail            // real validator
);

test('valid user returns 200', async () => {
  const response = await request(app)
    .post('/users')
    .send({
      username: 'Username',
      password: 'Password123',
      email: 'student@example.com'
    });

  expect(response.statusCode).toBe(200);
});

The Problem

Why Are These Tests So Slow?

Open validation/validateEmail.js and look at what it does:

function validateEmail(email) {
  // Intentionally slow — simulates a real-world external service call
  // (e.g. checking email validity against a remote API)
  const start = Date.now();
  while (Date.now() - start < 2000) {
    // busy-wait for 2 seconds on every call
  }

  if (!email || typeof email !== 'string') return false;
  const re = /^[^\s@]+@[^\s@]+\.[^\s@]{2,}$/i;
  return re.test(email);
}

module.exports = validateEmail;
Every test that calls the real validateEmail waits 2 seconds. 3 tests = 6 seconds minimum. A full suite of 30 tests = 1 minute just waiting.
In the real world this could be:
  • An external email verification API
  • A database lookup
  • A network call to a third-party service
  • A payment gateway
The real validateEmail is not what we are testing here — we are testing the API routing and validation wiring. We should not be blocked by it.
Mocking

What Is a Mock?

A mock replaces a real dependency with a fake one that you control — same interface, predictable behaviour, no side effects.

Why mock?
  • Remove slow dependencies (network, DB, timers)
  • Test in isolation — only your code under test
  • Simulate scenarios hard to reproduce (API errors, timeouts)
  • Make tests fast and deterministic
What can you mock?
  • Modules (jest.mock('./path'))
  • Functions (jest.fn())
  • Return values (mockReturnValue())
  • Implementations (mockImplementation())
MockVerifiable fake — records calls, can assert on them
StubReturns canned responses — doesn't record calls
SpyWraps the real function — records calls but still runs real code
jest.mock

Mocking a Module with Jest

jest.mock() intercepts require() calls and replaces the module with your fake version.

// app.mock.test.js

const createApp = require('./app');
const request   = require('supertest');
const validateUsername = require('./validation/validateUsername');
const validatePassword = require('./validation/validatePassword');

// Replace the slow real validateEmail with a fast fake
jest.mock('./validation/validateEmail', () => {
  return jest.fn((email) => {
    // No 2-second wait — just a fast regex check
    if (!email || typeof email !== 'string') return false;
    const re = /^[^\s@]+@[^\s@]+\.[^\s@]{2,}$/i;
    return re.test(email);
  });
});

const validateEmail = require('./validation/validateEmail'); // gets the mock
const app = createApp(validateUsername, validatePassword, validateEmail);
jest.fn(implementation) Creates a mock function. Tracks how many times it was called, with which arguments, and what it returned.
Important: jest.mock() is hoisted to the top of the file by Jest before any imports run — that is why it works even though it appears after the requires.
Mock Assertions

Verifying the Mock Was Called

Because jest.fn() records every call, you can assert on how a dependency was used — not just the outcome.

const validateEmail = require('./validation/validateEmail'); // the mock

test('validateEmail is called with the provided email', async () => {
  await request(app).post('/users').send({
    username: 'Username',
    password: 'Password123',
    email: 'student@example.com'
  });

  // Was it called at all?
  expect(validateEmail).toHaveBeenCalled();

  // Was it called with the right argument?
  expect(validateEmail).toHaveBeenCalledWith('student@example.com');
});

test('reset mock between tests', () => {
  // Clear call history before each test
  beforeEach(() => {
    validateEmail.mockClear();   // clear calls
    // validateEmail.mockReset(); // also clear return value / implementation
  });
});
Asserting on mock calls turns a black-box test into a white-box one — you verify the wiring, not just the output.
Comparison

Real Validators vs Mocked

app.test.js — Real validateEmail
  • Uses the real 2-second validator
  • 3 tests ≈ 6+ seconds to run
  • Tests the full stack end-to-end
  • Slow feedback loop in development
  • Fragile if the external service changes
app.mock.test.js — Mocked validateEmail
  • Replaces validator with a fast fake
  • 3 tests ≈ <200 ms to run
  • Tests routing & wiring in isolation
  • Fast feedback — runs on every save
  • You control what the mock returns
Both have value — they test different things. Use mocked tests for rapid development feedback. Keep a few real integration tests to verify the actual validator still works end-to-end.
Design Pattern

Why Dependency Injection Matters for Testing

The reason mocking is easy in User-Register is that app.js uses Dependency Injection.

// ❌ Hard to test — dependency is hidden inside
function createApp() {
  const validateEmail =
    require('./validation/validateEmail'); // baked in
  app.post('/users', (req, res) => {
    // can't swap validateEmail in tests
  });
}
Hard-coded dependency → you must mock the whole module OR live with the 2-second wait.
// ✅ Easy to test — dependency is injected
function createApp(
  validateUsername,
  validatePassword,
  validateEmail       // ← passed in from outside
) {
  app.post('/users', (req, res) => {
    // caller decides which validator to use
  });
}
Injected dependency → pass a real one in production, pass a mock in tests. No module-level hacking needed.
Testable code is usually better-designed code. If something is hard to test, it is often a signal to refactor.
Best Practices

When to Mock — and When Not To

✅ Good reasons to mock:
  • Dependency is slow (network, DB, timers)
  • Dependency is non-deterministic (random, time-based)
  • Dependency causes side effects (email, payments, SMS)
  • You want to test error paths that are hard to trigger
  • The dependency isn't built yet
❌ Avoid mocking when:
  • You end up mocking so much that the test proves nothing
  • The dependency is fast, local, and deterministic
  • You are testing the dependency itself
  • Mocking hides a real design problem in your code
A test that mocks everything is not testing your code — it is testing your mocks.
