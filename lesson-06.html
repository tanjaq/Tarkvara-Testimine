The Testing Pyramid

Where API Tests Fit

Unit tests cover individual functions. API tests cover the layer above — how components work together.

E2E / UI Tests
Integration / API Tests
Unit Tests
Integration tests verify:
  • Components work correctly together
  • HTTP routing and status codes are right
  • Request parsing and response format
  • Validation logic is correctly wired into the API
Trade-off vs unit tests: Slower to run, harder to isolate failures — but catch problems that unit tests miss: wiring, middleware, routing.
Introduction

What Is an API?

An API (Application Programming Interface) is a contract between two software systems — it defines what requests one system can make, and what responses it will get back.

Why APIs exist
  • Let systems talk without exposing internals
  • Reuse functionality across many clients
  • Enable third parties to build on your platform
  • Separate frontend from backend
How APIs are used
  • Integration — connect different systems (shop → payment provider)
  • Data access — expose data without giving DB access
  • Automation — systems interact without human involvement
Real-world examples
  • "Continue with Google" — OAuth API, no password handling
  • momondo.com — calls hundreds of airline APIs at once
  • Google Maps — navigation in thousands of third-party apps
  • Stripe — payment processing without storing card data
REST & Databases

CRUD Operations & HTTP Methods

REST maps directly onto the four basic database operations. Every REST endpoint is essentially one of these four things.

CRUD HTTP Method Database operation User-Register example Typical status
Create POST INSERT INTO ... POST /users 200 / 201
Read GET SELECT * FROM ... GET /users/{id} 200 / 404
Update PUT PATCH UPDATE ... SET ... PUT /users/{id} 200 / 404
Delete DELETE DELETE FROM ... DELETE /users/{id} 200 / 404
PUT vs PATCH PUT replaces the entire resource — send all fields. PATCH updates only the fields you include (partial update).
GET is safe & idempotent It never changes data. Calling it 100 times has the same result as once — important for test design.
HTTP Status Codes

Reading the Response Code

The status code is the first thing to assert in any API test — it tells you whether the server understood and processed the request correctly.

2xx — Success
  • 200 OK — request succeeded
  • 201 Created — resource was created
  • 204 No Content — success, no body
4xx — Client Error
  • 400 Bad Request — invalid input
  • 401 Unauthorised — missing/invalid token
  • 403 Forbidden — authenticated, no permission
  • 404 Not Found — resource doesn't exist
  • 422 Unprocessable — validation failed
5xx — Server Error
  • 500 Internal Server Error — server crashed
  • 502 Bad Gateway — upstream service failed
  • 503 Service Unavailable — overloaded
A 500 in response to bad user input is always a bug — the server should validate and return 400, never crash.

Mocking

What Is a Mock?

A mock replaces a real dependency with a fake one you control — same interface, predictable behaviour, no side effects.

Why mock?
  • Remove slow dependencies (network, DB, timers)
  • Test in isolation — only your code under test
  • Simulate scenarios hard to reproduce (API errors, timeouts)
  • Make tests fast and deterministic
What can you mock in Jest?
  • Whole modules — jest.mock('./path')
  • Individual functions — jest.fn()
  • Return values — mockReturnValue()
  • Implementations — mockImplementation()
MockVerifiable fake — records calls, lets you assert on them
StubReturns canned responses — doesn't record calls
SpyWraps the real function — records calls but still runs the real code
jest.mock

How jest.mock Works

jest.mock() intercepts require() and replaces a module with your fake before any test runs.

app.test.js Uses the real validateEmail — every call waits 2 seconds. Full stack, slow feedback.
app.mock.test.js jest.mock() swaps in a fast fake. Same scenarios, same assertions — runs in milliseconds.
jest.fn() Creates a mock function that tracks every call — how many times, with which arguments. Assert on behaviour, not just output.
Comparison

Real Validators vs Mocked

app.test.js — Real validateEmail
  • Uses the real 2-second validator
  • 3 tests ≈ 6+ seconds to run
  • Tests the full stack end-to-end
  • Slow feedback loop in development
  • Fragile if the external service changes
app.mock.test.js — Mocked validateEmail
  • Replaces validator with a fast fake
  • 3 tests ≈ <200 ms to run
  • Tests routing & wiring in isolation
  • Fast feedback — runs on every save
  • You control exactly what the mock returns
Both have value — they test different things. Use mocked tests for rapid development feedback. Keep a few real integration tests to verify the actual validator still works end-to-end.
Best Practices

When to Mock — and When Not To

✅ Good reasons to mock:
  • Dependency is slow (network, DB, timers)
  • Dependency is non-deterministic (random, time-based)
  • Dependency causes side effects (sends email, charges a card)
  • You want to test error paths that are hard to trigger for real
  • The dependency isn't built yet
❌ Avoid mocking when:
  • You end up mocking so much that the test proves nothing
  • The dependency is fast, local, and deterministic
  • You are testing the dependency itself
  • Mocking hides a real design problem in your code
A test that mocks everything is not testing your code — it is testing your mocks.
