APIs in the Real World

Where APIs Appear Every Day

An API (Application Programming Interface) is a contract that lets two systems talk to each other without knowing each other's internal details.

Universal logins"Continue with Google / Facebook" — your app calls their API, never touches the password.
Travel comparisonsSites like momondo.com call hundreds of airline APIs at once to find the cheapest flight.
MapsGoogle Maps API powers navigation in thousands of other apps — directions, speed limits, traffic.
Payment gatewaysStripe, PayPal — your shop calls their API. You never store a card number.
IoT devicesSmart home devices expose APIs so apps can read sensors and send commands.
Why this matters for testers: Every integration point is a place where things can break. API tests catch those breaks before users do.
API Types

API Architecture Styles

REST (Representational State Transfer) Uses HTTP methods on resource URLs. Stateless. Returns JSON or XML. The dominant style for web and mobile APIs today.
Used by: Twitter, Stripe, GitHub, PetStore
SOAP (Simple Object Access Protocol) XML-based, strict standards and contracts. Verbose but reliable. Common in banking, insurance, and enterprise legacy systems.
Used by: PayPal (legacy), many bank integrations, SAP
GraphQL One endpoint — client asks for exactly the fields it needs. No over- or under-fetching.
Used by: GitHub, Shopify, Facebook, Airbnb
Webhooks The server calls you when something happens. You register a URL; the provider sends a POST when an event fires — push, not pull.
Used by: GitHub (push events), Stripe (payments), Twilio (SMS)
gRPC (Google Remote Procedure Call) Binary protocol over HTTP/2. Very fast. Uses Protocol Buffers (not JSON). Not human-readable.
Used by: Google, Netflix, internal microservices
In this course → REST Most common, best tooling, easiest to test manually. Everything we do in Postman is REST.

API Test Checklist

What to Test in an API

Functional correctness
  • Status codes match the scenario (200, 201, 400, 404…)
  • Response body has the right structure and field values
  • Data sent is correctly saved and returned
Data validation
  • Required fields — missing field returns 400
  • Data types — string where number expected → error
  • Boundary values — max length strings, zero, negatives
Error handling
  • Invalid input returns a meaningful error message
  • Non-existent resource returns 404, not 500
  • Server errors don't leak internal stack traces
Authentication & authorisation
  • Unauthenticated request → 401
  • Authenticated but wrong role → 403
  • Valid token → access granted
A test suite that only tests the happy path misses most real-world bugs. Always test negative and edge cases too.
Test Strategy

Functional · Negative · Edge Case

Functional Testing Verify the endpoint does what it is supposed to do with valid input.

Example: POST /pet with valid data → 200, returns the pet with an ID
Negative Testing Send invalid, missing, or wrong input. Verify the API rejects it gracefully.

Example: GET /pet/{id} with id = "abc" → 400 or 404 with a useful message
Edge Case Testing Boundary values, unusual combinations, empty strings, very large IDs.

Example: POST /pet with an empty name "" — does the API accept or reject it?
Homework requirement: Your test suite must include all three types — at least one functional, one negative, and one edge-case test per endpoint you choose to cover.
API Documentation

Swagger / OpenAPI

OpenAPI is a standard for describing REST APIs in a machine-readable format (JSON or YAML). Swagger UI renders it as an interactive browser page you can test from.

What Swagger shows you:
  • All available endpoints and HTTP methods
  • Required and optional parameters
  • Request body schema with field types
  • Possible response codes and response schemas
  • Try-it-out — send real requests from the browser
# OpenAPI spec (simplified)
paths:
  /pet/{petId}:
    get:
      summary: Find pet by ID
      parameters:
        - name: petId
          in: path
          required: true
          schema:
            type: integer
      responses:
        '200':
          description: Successful
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Pet'
        '404':
          description: Pet not found
Swagger is your contract — it tells you what the API should do. Your tests verify it actually does it.

Postman

What Is Postman?

Postman is the most widely-used API testing tool. It lets you build, send, and automate HTTP requests without writing code — and add test scripts when you're ready to go further.

Requests Build any HTTP request — method, URL, headers, body — and send it with one click.
Collections Group related requests into a folder. Run the whole collection with one click or from CI.
Test Scripts Write JavaScript assertions that run after every response. Automate your checks.
Environments Store variables (base URL, tokens) per environment — switch between dev / staging / prod.
Newman Command-line runner for Postman collections. Run your tests in CI/CD pipelines.
Download: postman.com/downloads

Free for individuals. No account required for basic use.

Common mistakes:
  • Forgetting Content-Type: application/json header
  • Sending malformed JSON (trailing comma, missing quotes)
  • Using Body → form-data instead of raw JSON
Postman — Test Scripts

Automating Assertions

The Tests tab runs JavaScript after the response arrives. Use pm.test() to write assertions.

// Tests tab — runs after every response

// 1. Check status code
pm.test("Status code is 200", function () {
    pm.response.to.have.status(200);
});

// 2. Response time
pm.test("Response time under 2000ms", function () {
    pm.expect(pm.response.responseTime).to.be.below(2000);
});

// 3. Response body structure
pm.test("Response has id and name", function () {
    const body = pm.response.json();
    pm.expect(body).to.have.property("id");
    pm.expect(body).to.have.property("name");
});

// 4. Field value
pm.test("Pet status is available", function () {
    const body = pm.response.json();
    pm.expect(body.status).to.eql("available");
});

// 5. Save to environment for next request
pm.environment.set("petId", pm.response.json().id);
Postman — Negative Tests

Testing Error Responses

Negative tests verify the API rejects bad input correctly — not just that the happy path works.

// GET /pet/99999999 — non-existent ID

pm.test("Returns 404 for unknown pet", function () {
    pm.response.to.have.status(404);
});

pm.test("Error message is present", function () {
    const body = pm.response.json();
    pm.expect(body).to.have.property("message");
});
// GET /pet/abc — invalid (non-numeric) ID

pm.test("Returns 400 or 404 for invalid ID", function () {
    pm.expect(pm.response.code).to.be.oneOf([400, 404]);
});
What to assert in error responses:
  • Status code is not 200
  • Error message field exists
  • No sensitive data in the error body
  • Server does not return 500 for client mistakes
Tip — use environment variables: Set {{baseUrl}} = https://petstore.swagger.io/v2 in your environment so your requests look like {{baseUrl}}/pet/{{petId}} instead of hard-coded URLs.
Postman — Collections

Organising Tests into a Collection

A Collection groups related requests and lets you run them all in sequence — like a test suite for your API.

GET Verify update

📁 Negative Tests
GET Non-existent ID → 404
GET Invalid ID (abc) → 4xx
Running a Collection:
  • Click Run collection → opens Collection Runner
  • Choose environment, number of iterations
  • Results show pass/fail per test per request
  • Export results as JSON or HTML report
Newman (CLI runner): 
npm install -g newman
newman run PetStore.postman_collection.json \
  -e PetStore.environment.json
Runs your Postman collection in CI/CD — no GUI needed.
