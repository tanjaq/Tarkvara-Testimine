Introduction

What Is an API?

An API (Application Programming Interface) is a contract between two software systems — it defines what requests one system can make, and what responses it will get back.

The analogy A restaurant menu is an API. You don't need to know how the kitchen works — you just pick from what's offered, and you get back a defined result.
Why APIs exist
  • Let systems talk without exposing internals
  • Reuse functionality across many clients
  • Enable third parties to build on your platform
  • Separate frontend from backend
Client → API → Server
  • Client sends a request (what it wants)
  • API routes the request to the right logic
  • Server processes it and returns a response
  • Client doesn't care how the server works internally
What crosses the wire URL + HTTP method + optional headers + optional body (JSON) → response with status code + body
API Usage

How APIs Are Used

Integration Connect different systems and services. A web shop integrates with a payment provider via API — no direct database access, just a defined contract.

Examples: Stripe, PayPal, SendGrid
Access to data Companies expose APIs so developers can build on their data without getting direct database access.

Examples: Twitter/X API, OpenWeather, Google Maps
Automation Software components interact and perform tasks without human intervention — CI/CD, monitoring, scheduled jobs.

Examples: GitHub Actions, Slack bots, Zapier
Universal logins "Continue with Google / Facebook" — your app calls their OAuth API. You never see the user's password.
IoT devices Smart home devices expose APIs so mobile apps and cloud services can read sensor data and send commands without knowing the hardware.
API Types

API Architecture Styles

REST (Representational State Transfer) Uses HTTP methods on resource URLs. Stateless. Returns JSON or XML. The dominant style for web and mobile APIs today.
Used by: Twitter, Stripe, GitHub, PetStore
SOAP (Simple Object Access Protocol) XML-based, strict standards and contracts (WSDL). Verbose but reliable. Common in banking, insurance, and enterprise legacy systems.
Used by: PayPal (old), many bank integrations, SAP
GraphQL One endpoint, client asks for exactly the fields it needs. No over- or under-fetching. Defined by a schema.
Used by: GitHub, Shopify, Facebook, Airbnb
Webhooks The server calls you when something happens — push, not pull. You register a URL; the provider sends a POST when an event fires.
Used by: GitHub (push events), Stripe (payment events), Twilio (SMS delivery)
gRPC (Google Remote Procedure Call) Binary protocol over HTTP/2. Extremely fast and efficient. Uses Protocol Buffers (not JSON). Not human-readable.
Used by: Google internal services, Netflix, microservice communication
In this course → REST Most common, best tooling support, easiest to test manually. Everything we do in Postman is REST.
REST & Databases

CRUD Operations & HTTP Methods

REST maps directly onto the four basic database operations. Every REST endpoint is essentially one of these four things.

CRUD HTTP Method Database operation PetStore example Typical status
Create POST INSERT INTO ... POST /pet 200 / 201
Read GET SELECT * FROM ... GET /pet/{id} 200 / 404
Update PUT PATCH UPDATE ... SET ... PUT /pet 200 / 404
Delete DELETE DELETE FROM ... DELETE /pet/{id} 200 / 404
PUT vs PATCH PUT replaces the entire resource (send all fields). PATCH updates only the fields you send (partial update).
GET is safe & idempotent It never changes data. Calling it 100 times has the same result as calling it once — important for test design.
HTTP Status Codes

Reading the Response Code

The status code is the first thing to assert in any API test — it tells you whether the server understood and processed the request.

2xx — Success
  • 200 OK — request succeeded
  • 201 Created — resource created
  • 204 No Content — success, no body
4xx — Client Error
  • 400 Bad Request — invalid input
  • 401 Unauthorised — no/invalid token
  • 403 Forbidden — authenticated but no permission
  • 404 Not Found — resource doesn't exist
  • 422 Unprocessable — validation failed
5xx — Server Error
  • 500 Internal Server Error — something crashed
  • 502 Bad Gateway — upstream failed
  • 503 Service Unavailable — server overloaded
A 500 in response to bad user input is always a bug — the server should validate and return 400, never crash.
API Test Checklist

What to Test in an API

Functional correctness
  • Status codes match the scenario
  • Response body has the right structure and field values
  • Data sent is correctly saved and returned
Data validation
  • Required fields — missing field returns 400
  • Data types — wrong type returns an error
  • Boundary values — zero, negatives, max length
Error handling
  • Invalid input returns a meaningful error message
  • Non-existent resource returns 404, not 500
  • Server errors don't leak internal stack traces
Authentication & authorisation
  • Unauthenticated request → 401
  • Authenticated but wrong role → 403
  • Valid token → access granted
A test suite that only tests the happy path misses most real-world bugs. Always cover negative and edge cases too.
Test Strategy

Functional · Negative · Edge Case

Functional Testing Verify the endpoint does what it is supposed to do with valid input.

POST /pet with valid data → 200, returns the pet with an ID
Negative Testing Send invalid, missing, or wrong input. Verify the API rejects it gracefully.

GET /pet/{id} with id = "abc" → 400 or 404 with a useful message, not a crash
Edge Case Testing Boundary values, unusual combinations, empty strings, very large numbers.

POST /pet with name = "" — does the API accept or reject an empty name?

API Documentation

Swagger / OpenAPI

OpenAPI is a standard for describing REST APIs in a machine-readable format (JSON or YAML). Swagger UI renders it as an interactive browser page.

What Swagger shows you:
  • All available endpoints grouped by resource
  • HTTP method for each endpoint
  • Required and optional parameters (path, query, header)
  • Request body schema with field names and types
  • All possible response codes and response schemas
  • Try it out — send real requests from the browser
Swagger as a requirements document
  • The spec says what the API should do
  • Your tests verify it actually does it
  • If behaviour doesn't match the spec → that is a bug
  • If the spec is wrong → raise it with the team
Who writes the spec? Can be generated from code (annotations), or written first as a design contract (spec-first). Many teams publish it publicly so external developers can build integrations.
Swagger is your contract. Test against it the same way you test against functional requirements.

Postman

What Is Postman?

The most widely-used API testing tool. Build, send, and automate HTTP requests — no code required to get started.

Requests Build any HTTP request — choose method, enter URL, set headers and body — then send with one click and inspect the full response.
Collections Group related requests into folders. Run the whole collection in sequence — like a test suite. Share and export as JSON.
Test Scripts Write JavaScript in the Tests tab. Assertions run automatically after each response — pass/fail reported per request.
Environments Store variables (base URL, token, IDs) per environment. Switch between dev/staging/prod without editing requests.
Newman CLI runner for collections. Run your Postman tests in a terminal or CI/CD pipeline without opening the GUI.
Get started:
postman.com/downloads
Free for individual use.
Postman — Requests

Making a Request in Postman

Every request in Postman has the same structure — method, URL, and optional tabs for headers, body, and tests.

GET {{baseUrl}}/pet/{{petId}} Send
Params Headers Authorization Body Pre-request Script Tests
No params needed — petId is a path variable defined in the URL above
Response panel shows: Status code · Response time · Body (JSON) · Headers · Size
For POST/PUT requests: Switch to Body tab → select raw → choose JSON → paste your JSON object
Environment variables: Use {{baseUrl}} and {{petId}} instead of hard-coded values — set once, reuse everywhere
Postman — Test Scripts

Writing Assertions in the Tests Tab

The Tests tab runs JavaScript after each response. Results appear as pass/fail in the test results panel.

Params Headers Body Tests
// Four common assertions — shown live during class
1. Check status code
2. Check response time
3. Check that a field exists in the body
4. Check a specific field value
// Save a value to use in the next request
5. Store the returned ID in an environment variable
pm.test(name, fn) Wraps one assertion. If it throws, the test fails. The name appears in the results panel — make it descriptive.
pm.expect() vs pm.response.to.have Both work — pm.expect() is Chai-style and more flexible. pm.response.to.have.status(200) is a shorthand for the most common check.
Postman — Negative Tests

Testing Error Responses

Create separate requests for failure scenarios — give them clear names so the collection reads like a test plan.

GET {{baseUrl}}/pet/99999999
Test: status is 404 · message field exists
GET {{baseUrl}}/pet/abc
Test: status is 4xx · server did not return 500
What to assert in error responses:
  • Status code is the expected error code (not 200)
  • Error message field exists and is readable
  • No sensitive data (stack traces, DB errors) in the body
  • Server returns a 4xx for client mistakes — never 5xx
Tip — chain requests: POST a pet, save its ID, then DELETE it, then GET it to confirm 404. This tests the full lifecycle in one collection run.
Postman — Collections

Organising Tests into a Collection

A Collection groups requests into folders and runs them in order — each with its own test assertions.

📁 PetStore Tests
📁 Happy Path
POST Create pet → save ID
GET Get pet by ID → verify name
PUT Update pet name
GET Verify update persisted
DEL Delete pet
GET Confirm 404 after delete
📁 Negative Tests
GET Non-existent ID → 404
GET Invalid ID format → 4xx
Running a Collection:
  • Click Run collection → Collection Runner opens
  • Select environment, set iteration count
  • Results: pass/fail per test per request
  • Export as JSON or HTML report
Newman — run in terminal:

Install: npm install -g newman
Run: newman run collection.json -e env.json

No GUI needed — plug into CI/CD pipelines.
